## 0. 心智模型

把工具系统看成**给 LLM 用的 SDK**，而不是普通的内部函数。区别：

| 普通函数 SDK | LLM 工具 SDK |
|---|---|
| 调用者是人，会读文档 | 调用者是模型，只读 description |
| 报错给开发者看 | 报错是给模型的**下一轮 prompt** |
| 鼓励灵活性、可选参数多 | 可选越多模型越乱填，要克制 |
| 失败抛异常即可 | 失败必须能让模型**自我纠正** |
| 命名风格统一即可 | 命名要"可预测"，符合模型先验 |
| 参数校验防御 bug | 参数校验防御**幻觉** |

**一句话**：工具的设计质量 = Agent 的可靠性上限。模型再强，工具描述模糊、错误信息糟糕，照样翻车。

## 1. Schema 工程（Schema Engineering）

工具 Schema = `name + description + input_schema (JSON Schema)`。这三样里，**description 比 name 重要 10 倍**。

### 1.1 Description 写作规范

**坏例子**：

```json
{
  "name": "search",
  "description": "Search for things.",
  "input_schema": { ... }
}
```

模型完全不知道：搜什么？返回什么格式？什么时候用它？什么时候不该用？

**好例子**：

```json
{
  "name": "web_search",
  "description": "Search the public web via Tavily and return the top 5 results as a list of {title, url, snippet}. Use this when the user asks about current events, recent papers, or anything that may have changed after your training cutoff. Do NOT use this for questions answerable from prior conversation context or from `search_my_notes`. Each call costs ~$0.005 and takes 2-4s; prefer one focused query over multiple broad ones.",
  "input_schema": { ... }
}
```

**Description 必须回答 5 个问题**：

1. **做什么**（What）：一句话能复述
2. **返回什么格式**（Returns）：模型要据此规划下一步
3. **何时使用**（When to use）：触发场景
4. **何时不使用**（When NOT to use）：与其他工具的边界
5. **代价**（Cost / Latency）：让模型权衡是否多调

> **一个工程口诀**：Description 是给模型读的"工具卡片"。把它当成 Stripe API 文档的开头那段写。

### 1.2 参数命名："可预测性" > "精确性"

模型有强先验，命名违反先验它就会写错。

| 反例 | 正例 | 原因 |
|---|---|---|
| `file_location` | `path` | "path" 是模型见过 100 万次的命名 |
| `search_string` | `query` | "query" 是搜索 API 的标准词 |
| `target_user_identifier` | `user_id` | 简短 + 标准 |
| `should_include_archived` | `include_archived` | 布尔不要加 "should_" |
| `mode` (可选 5 个值) | `read_only: bool` | 多枚举不如多布尔清晰 |

**命名原则**：

1. **跟随主流 SDK 的词汇**：path、query、id、limit、offset、cursor、since、until
2. **布尔参数用 `is_/has_/include_` 前缀**，不用 `should_`
3. **避免缩写**（`max_results` 不用 `max_n`），除非是行业惯例（`url`、`id`）
4. **类型暗示在名字里**：`*_at`（时间戳）、`*_ids`（列表）、`*_count`（整数）

### 1.3 必需 vs 可选参数的取舍

**铁律**：可选参数越多，模型越容易胡乱填。

每个可选参数都是一次"幻觉机会"。

**判断准则**：

- 这个参数 99% 的调用都用默认值？→ **删掉，写死在工具内部**
- 这个参数有 2-3 个常见值？→ **拆成多个工具**（`search_recent` / `search_all`）比加一个 `time_range` 参数好
- 这个参数模型大概率不知道该填什么？→ **改成必需 + 在 description 给指导**

**反例**：

```python
def search(
    query: str,
    max_results: int = 5,
    include_images: bool = False,
    search_depth: Literal["basic", "advanced"] = "basic",
    domains: list[str] | None = None,
    exclude_domains: list[str] | None = None,
    time_range: Literal["day", "week", "month", "year"] | None = None,
)
```

模型会面对 7 个参数手足无措。

**正例**：

```python
def web_search(query: str) -> list[SearchResult]:
    """Search public web. Top 5 results."""
    # max_results=5、search_depth="basic" 写死
```

```python
def web_search_recent(query: str, since_days: int) -> list[SearchResult]:
    """Search public web limited to results from the past N days. Use for current events."""
```

把 `time_range` 拆成独立工具，比塞 7 参数清楚得多。

### 1.4 JSON Schema 的工程化用法

**用 Pydantic 自动生成 schema**：

```python
from pydantic import BaseModel, Field

class WebSearchInput(BaseModel):
    query: str = Field(
        description="Concise search query, 3-10 words. Avoid full sentences. Example: 'GPT-5 release date' not 'When will GPT-5 be released?'"
    )

def to_anthropic_tool(model_cls, name, description) -> dict:
    schema = model_cls.model_json_schema()
    return {
        "name": name,
        "description": description,
        "input_schema": {
            "type": "object",
            "properties": schema["properties"],
            "required": schema.get("required", []),
        },
    }
```

**Field description 也是给模型看的**。每个参数都写一句"怎么填 + 一个例子"。

**枚举优于自由文本**：能用 `Literal["read", "write"]` 就别用 `str`。

**用 `pattern`/`minimum`/`maxItems` 做硬约束**：模型会读这些约束。

## 2. 错误处理哲学

> **工具的报错信息是给模型看的下一轮 prompt。**

这句话改变一切。错误信息不是日志，是**指令**。

### 2.1 三种错误响应风格对比

**风格 A（开发者风格，最差）**：

```
FileNotFoundError: [Errno 2] No such file or directory: 'foo.txt'
Traceback (most recent call last):
  File "tool.py", line 42, in read_file
    ...
```

模型能看懂这是文件不存在，但不知道下一步该做什么。多数会重试同样的调用。

**风格 B（友好但无指导）**：

```
Error: File not found.
```

模型仍然不知道下一步。

**风格 C（教练风格，最好）**：

```
Error: File 'foo.txt' not found in cwd '/work/project'.

Suggestions:
- Use `list_dir(".")` to see available files
- Check if you meant 'foo.md' (similar name found)
- The path is relative to the project root, not absolute
```

模型 90% 的概率会在下一轮调对。

### 2.2 错误响应的设计模板

每个错误返回都包含 3 部分：

```python
@dataclass
class ToolError:
    what_happened: str          # 客观事实
    why_it_failed: str          # 可能原因（如能推断）
    what_to_try_next: list[str] # 可执行的修复建议
```

序列化到模型时变成：

```
[Tool error: read_file]
What: File 'foo.txt' not found.
Why: The path may be relative to a different cwd, or the file may have a different extension.
Try:
  1. Call list_dir(".") to see available files.
  2. Did you mean 'foo.md'? (close match)
  3. If you intended an absolute path, prefix with '/'.
```

### 2.3 常见错误类型的处理范式

| 错误类型 | 标准应对 |
|---|---|
| **参数校验失败** | 指出**哪个参数**错了 + 期望格式 + 一个正确示例 |
| **资源不存在** | 列出**相近的可选项**（fuzzy match） |
| **权限/危险操作** | 明确说"此操作被禁止"+ 说**为什么** + 替代方案 |
| **超时** | 告诉它超时阈值是多少 + 建议拆分任务或缩小范围 |
| **外部 API 失败** | 区分是临时（建议重试）还是永久（不要重试） |
| **结果为空** | 不要返回空数组就完事，明确说"零结果"+ 建议放宽条件 |
| **结果过多** | 截断 + 告知总数 + 建议加过滤条件 |

### 2.4 重试策略

**模型层面**：错误信息里**显式说"可重试"或"不要重试"**，否则模型乱重试。

**工具层面**：

```python
@retry(
    retries=3,
    backoff=lambda i: 2**i,
    retryable=lambda e: isinstance(e, (TimeoutError, RateLimitError)),
)
def call_external_api(...): ...
```

**死循环检测**：Agent 主循环里维护"最近 N 轮的工具调用签名"，同一签名重复 ≥ 3 次直接打断并提示模型换思路。

```python
def detect_loop(history: list[ToolCall], window: int = 5, threshold: int = 3) -> bool:
    recent = history[-window:]
    sigs = [(t.name, hash_args(t.input)) for t in recent]
    return any(sigs.count(s) >= threshold for s in sigs)
```

## 3. 危险操作隔离

任何工具一旦能**修改外部状态**（写文件、执行命令、调付费 API、发消息），就需要独立的安全设计。

### 3.1 风险分级

| 级别 | 例子 | 策略 |
|---|---|---|
| **L0 只读** | `read_file`、`web_search`、`grep` | 直接放行，记日志 |
| **L1 局部写** | `write_file`（沙箱目录内） | 路径白名单 + 大小上限 |
| **L2 系统写** | `run_python`、`shell` | 沙箱 + 超时 + 资源限制 |
| **L3 不可逆 / 钱** | `git push`、`send_email`、`pay_api` | 默认禁用 + 显式确认 + 审计日志 |

**默认拒绝原则**：拿不准放哪一级 → 放高一级。

### 3.2 路径白名单 / 黑名单

**白名单优于黑名单**（黑名单永远不全）：

```python
ALLOWED_ROOTS = [Path("/work/sandbox").resolve()]

def safe_resolve(user_path: str) -> Path:
    p = (ALLOWED_ROOTS[0] / user_path).resolve()
    if not any(p.is_relative_to(root) for root in ALLOWED_ROOTS):
        raise ToolDenied(
            f"Path '{user_path}' resolves outside the sandbox. "
            f"Allowed roots: {[str(r) for r in ALLOWED_ROOTS]}. "
            f"All paths must be within /work/sandbox."
        )
    return p
```

**陷阱**：必须 `resolve()` 之后再判断，否则 `../../../etc/passwd` 绕过白名单。

**黑名单做兜底**：禁止后缀 `.env`、`id_rsa`、`*.key`、`/proc/`、`/sys/` 等。

### 3.3 子进程沙箱

```python
import subprocess

def run_python(code: str, timeout_s: int = 10) -> ToolResult:
    try:
        proc = subprocess.run(
            ["python", "-c", code],
            capture_output=True,
            text=True,
            timeout=timeout_s,
            cwd="/work/sandbox",
            env={"PATH": "/usr/bin:/bin", "PYTHONDONTWRITEBYTECODE": "1"},
        )
    except subprocess.TimeoutExpired:
        return ToolError(
            what_happened=f"Execution exceeded {timeout_s}s timeout.",
            why_it_failed="The code may have an infinite loop or expensive computation.",
            what_to_try_next=[
                "Add print statements to find the slow part",
                "Break the task into smaller steps",
            ],
        )
    return ToolResult(stdout=proc.stdout[:10_000], stderr=proc.stderr[:2_000], returncode=proc.returncode)
```

**关键控制点**：

1. **`timeout` 必须设**（防止 fork bomb / 死循环）
2. **`cwd` 限定**到沙箱
3. **`env` 清洗**（只保留必要的）
4. **输出截断**（不让 100MB stdout 撑爆 context）
5. **`returncode` 透传**（让模型知道是否成功）
6. **网络隔离**（生产中用 Docker `--network=none` 或 firewall）

**升级路径**：本地学习用 subprocess + cwd；生产用 Docker / gVisor / Firecracker / E2B / Modal。

### 3.4 确认机制（Human-in-the-loop）

L3 级别的工具不应"调用即执行"，而是：

```python
def send_email(to: str, subject: str, body: str) -> ToolResult:
    if not _user_confirmed(operation="send_email", payload={"to": to, "subject": subject}):
        return ToolError(
            what_happened="Action requires user confirmation. Awaiting approval.",
            why_it_failed="Sending email is irreversible and was flagged as L3.",
            what_to_try_next=[
                "Summarize what you plan to send and ask the user to confirm",
                "Or use draft_email() to save without sending",
            ],
        )
    return _send_real(to, subject, body)
```

实现方式可以是：

- **CLI 模式**：在终端弹 `[y/N]` 提问
- **Web 模式**：返回 "pending_approval" 状态，前端展示卡片让用户点确认
- **批准列表**：用户预批准 "本会话内允许发邮件给 a@b.com"

### 3.5 审计日志

每次 L1+ 调用都要持久化记录：

```python
@dataclass
class ToolAuditEntry:
    ts: datetime
    session_id: str
    tool_name: str
    input_redacted: dict     # 敏感字段（密码、token）做脱敏
    output_summary: str
    risk_level: str
    user_confirmed: bool
```

学习阶段用 SQLite 即可。生产用结构化日志 + Langfuse trace。

## 4. 并行与顺序执行

Anthropic / OpenAI 都支持模型在**单轮内返回多个 `tool_use`**，可以并行执行。

### 4.1 何时并行 vs 何时顺序

| 场景 | 模式 |
|---|---|
| 多个**独立查询**（搜 3 个不同关键词） | **并行** |
| 后一步依赖前一步结果（先 grep 再 read） | **顺序** |
| 写操作 / 副作用 | **顺序**（除非确定无冲突） |
| 同一资源的读 + 写 | **顺序** |

**让模型知道"可以并行"**：在 system prompt 加一句：

```
When the next steps are independent (e.g. searching multiple unrelated keywords),
return all tool calls in a single turn so they can run in parallel. Use sequential
calls only when one tool's output feeds into another's input.
```

### 4.2 调度器实现（最小版）

```python
import asyncio

async def execute_tool_calls(calls: list[ToolUse], tools: dict, max_concurrency: int = 5):
    sem = asyncio.Semaphore(max_concurrency)

    async def run_one(call: ToolUse) -> ToolResult:
        async with sem:
            try:
                return await asyncio.wait_for(
                    tools[call.name].run(**call.input),
                    timeout=tools[call.name].timeout_s,
                )
            except asyncio.TimeoutError:
                return ToolError(what_happened=f"Tool {call.name} timed out", ...)
            except Exception as e:
                return ToolError(what_happened=str(e), ...)

    results = await asyncio.gather(*[run_one(c) for c in calls], return_exceptions=False)
    return results
```

**要点**：

1. **`asyncio.gather` + Semaphore** 限制并发数（防止把外部 API 打挂）
2. **`return_exceptions=False`** 但单个工具内部要捕获所有异常并转成 `ToolError`，否则一个失败拖垮全部
3. **每个工具有独立的 timeout**（不同工具耗时差异大）
4. **结果顺序对应输入顺序**（`tool_use_id` 必须能配对）

### 4.3 顺序执行 + 早停

模型有时返回多个调用，但其实**第一个失败后续没必要做**（比如 read_file 失败再 grep 它没意义）。可在调度器加个 `fail_fast` 选项：

```python
async def execute_sequential(calls, tools, fail_fast: bool = True):
    results = []
    for call in calls:
        r = await run_one(call)
        results.append(r)
        if fail_fast and isinstance(r, ToolError):
            # 把"由于前一步失败，跳过了 X、Y"也告诉模型
            for skipped in calls[len(results):]:
                results.append(ToolError(
                    what_happened=f"Skipped {skipped.name} because previous tool failed.",
                    ...
                ))
            break
    return results
```

但默认建议不用 fail_fast，因为模型可能就指望并行调用里某个失败、某个成功的混合结果。

### 4.4 资源冲突保护

如果两个并行调用都写同一个文件 → 数据损坏。

**最小方案**：在工具层加 per-resource 锁：

```python
_locks: dict[str, asyncio.Lock] = defaultdict(asyncio.Lock)

async def write_file(path: str, content: str):
    async with _locks[path]:
        ...
```

**更好方案**：调度器层做静态分析 —— 检测一批 tool_use 是否有写冲突，有就强制串行。学习阶段不用做这么复杂，但要意识到这个问题存在。

## 5. 工具层的可观测性

每次工具调用都该产生一条结构化记录：

```python
@dataclass
class ToolTrace:
    tool_name: str
    input: dict
    output_summary: str
    duration_ms: int
    cost_usd: float | None
    success: bool
    error_type: str | None
    parallel_group_id: str | None
```

**最少要看的 4 个指标**：

1. **每个工具的调用频次**（哪些被滥用）
2. **每个工具的成功率**（哪些经常失败）
3. **每个工具的 p50/p95 延迟**（瓶颈在哪）
4. **每会话的工具调用次数分布**（是否陷入循环）

W11 接 Langfuse 时，把这些 trace 自动上报。

## 7. 反模式速查

| 反模式 | 后果 | 改法 |
|---|---|---|
| description 写一句"做某事" | 模型不会用 / 乱用 | 答 §1.1 的 5 问 |
| 7 个可选参数 | 模型乱填 | 拆工具 / 写死默认值 |
| 抛 Python Traceback | 模型读不懂 | 用 ToolError 三段式 |
| 黑名单防越权 | 永远不全 | 白名单 + resolve |
| 没 timeout 的 subprocess | 卡死 / fork bomb | timeout=10s 起 |
| 没并发上限的 gather | API 限流 / 打挂下游 | Semaphore |
| 静默吞异常 | 模型以为成功 | 显式返回 ToolError |
| 工具内部 retry 但不告诉模型 | 上下文里看不出延迟来源 | trace 里记重试次数 |
| 工具返回 100MB 内容 | 上下文爆炸 | 截断 + 给 expand 工具 |
