## W1 · Agent 基础范式与 Tool Use

### 学习目标
- 理解 ReAct、Function Calling、Tool Use 三种范式的差异与演化
- 掌握 Anthropic / OpenAI 两家的 tool calling 协议字段
- 能手写一个最小可用的 agent loop，理解 `stop_reason` 的含义

### 理论内容
1. **Agent 定义**：LLM + 工具 + 循环 + 终止条件。注意区分 "agent" 与 "workflow"（Anthropic 2024 博文 *Building effective agents*）。
2. **范式演进**：
   - ReAct（Yao et al. 2022）：Thought / Action / Observation 文本协议
   - Function Calling（OpenAI 2023）：把 Action 升级为结构化 JSON
   - Tool Use（Anthropic）：标准化 `tool_use` / `tool_result` 块
3. **核心循环**：
   ```python
   while response.stop_reason == "tool_use":
       response = LLM(messages, tools)
       results = execute_tools(response)
       messages.append(results)
   ```
4. **常见 stop_reason**：`end_turn` / `tool_use` / `max_tokens` / `stop_sequence` —— 各自的处理策略。

### 实践任务
- [ ] 阅读 `agents/s01_agent_loop.py` 源码，逐行注释
- [ ] 用 `<100` 行复现一个支持单个 `bash` 工具的 agent
- [ ] 故意制造死循环（让模型反复调用工具），观察 `max_tokens` 和迭代上限的保护机制
- [ ] 切换至少 2 个模型（如 `claude-sonnet-4-6` 与一个开源模型），对比 tool-call 稳定性

### 推荐阅读
- Anthropic *Building effective agents* (2024)
- OpenAI *Function calling guide*
- 论文：*ReAct: Synergizing Reasoning and Acting in Language Models*

## W2 · 工具系统设计

### 学习目标
- 设计可扩展的工具抽象（基类、参数 schema、注册表、分发器）
- 理解 schema 自动生成、参数校验、错误恢复三大要点
- 学会"工具即接口"的思维：把外部世界封装为模型可调用的原语

### 理论内容
1. **工具抽象层次**：
   - L0：纯函数 `(str) -> str`
   - L1：带 schema 的函数（自动生成 OpenAI/Anthropic schema）
   - L2：Tool 类（生命周期、状态、权限）
   - L3：可展开工具（一个类导出多个 sub-tool，如 memory_add / memory_search）
2. **Schema 工程**：
   - 描述比名字重要 10 倍（模型主要靠 description 决定是否调用）
   - 参数命名要"可预测"（`path` 比 `file_location` 好）
   - 必需 vs 可选参数的取舍——可选越多，模型越容易胡乱填
3. **错误处理哲学**：工具的报错信息是给模型看的"提示"，要写得能让模型自我纠正（"文件不存在，请先用 ls 查看"比"FileNotFoundError"好）
4. **危险操作隔离**：黑名单、白名单、确认机制、沙箱（subprocess 超时 / cwd / 资源限制）

### 实践任务
- [ ] 仿照本仓库 `tools/base.py` 实现一个 `Tool` 基类，支持 `to_cc_schema()` / `to_openai_schema()`
- [ ] 实现 `ToolRegistry`，支持注册、注销、批量分发
- [ ] 内置 5 个工具：`bash`、`read`、`write`、`edit`、`http_get`（可对照 `tools/builtin/` 中的实现）
- [ ] 写一个"故意错误"测试集：缺参数、错类型、空字符串、超大输入，观察 agent 的恢复行为
- [ ] 进阶：用 `@tool_action` 装饰器实现可展开工具（参考 `tools/base.py:AutoGeneratedTool`）

### 设计 checklist
```
□ 工具名称用动词或动宾结构，全小写蛇形
□ description 第一句说"做什么"，第二句说"什么时候用"
□ 必需参数 ≤ 3 个；超过就考虑拆分工具
□ 返回值始终是 str，且 ≤ 50KB（防止爆上下文）
□ 危险操作有黑名单 + 超时
□ 失败信息包含可执行的下一步建议
```

## W3 · 上下文工程（Context Engineering）

### 学习目标
- 理解上下文窗口是 agent 的"工作记忆"，而非"无限磁盘"
- 掌握 system prompt 分层、few-shot 注入、压缩、裁剪四类核心手段
- 能为 agent 设计一个自适应的上下文管理策略

### 理论内容
1. **上下文预算**：
   - 输入 token 直接影响延迟与成本；输出 token 决定 max_tokens
   - 经验法则：留出 30% 给"未来的工具结果与回复"
2. **System Prompt 分层结构**（推荐顺序）：
   1. 角色与目标
   2. 能力与约束（"你能做什么、不能做什么"）
   3. 工具使用准则（何时调用、何时不调用）
   4. 输出格式规范
   5. 安全红线
3. **Prompt Caching**（Anthropic）：
   - 把稳定的部分（system + 工具定义 + 长文档）放在前面并打 `cache_control` 标记
   - 缓存命中可降低 90% 成本、3-4 倍速度提升
   - 结构：稳定前缀 → 可变后缀
4. **上下文压缩策略**：
   - **滚动窗口**：保留最近 N 轮 + 永远保留最初系统/任务消息
   - **摘要压缩**：超过阈值时，让 LLM 把旧消息压成摘要插回
   - **重要性采样**：只保留含工具结果或用户决策的关键消息
5. **结构化输出**：用 XML 标签 / JSON schema 锁定输出格式，比"请用 JSON 回复"稳定得多

### 实践任务
- [ ] 给已有 agent 加 prompt caching，用 API 返回的 `cache_read_input_tokens` 验证命中
- [ ] 实现一个 `compress(messages)` 函数：超过 50K token 时，把前一半压成摘要
- [ ] 写一个 system prompt 模板生成器：传入角色、工具列表、约束，输出结构化 prompt
- [ ] A/B 测试：同一任务在「无压缩 / 滑窗 / 摘要压缩」三种策略下的准确率与成本

### 关键概念地图
```
上下文窗口
├── System Prompt (cached)
├── Tools Schema (cached)
├── Few-shot Examples (cached)
├── 长期记忆 / 检索结果 (semi-stable)
├── 对话历史 (rolling window + 摘要)
└── 当前用户输入 (volatile)
```

## W4 · 记忆管理

### 学习目标
- 理解 agent 记忆的四层模型，能为不同场景选择正确的存储方式
- 掌握"显式记忆 vs 隐式上下文"的边界
- 会设计 memory 的写入策略、检索策略、衰减策略

### 理论内容
1. **四层记忆模型**：
   | 层 | 生命周期 | 存储介质 | 典型内容 |
   |----|---------|----------|----------|
   | 工作记忆 | 当前请求 | 上下文窗口 | 当前轮次的工具结果 |
   | 短期记忆 | 当前会话 | 内存/文件 | 多轮对话状态、todo 列表 |
   | 长期记忆 | 跨会话 | 向量库/KV | 用户偏好、历史决策 |
   | 程序性记忆 | 永久 | 代码/skill 文件 | 工作流、领域知识 |

2. **写入时机**（什么值得记？）：
   - 用户明确说"记住 X"
   - 用户纠正了 agent 的行为（feedback）
   - 反复出现的模式（事实型）
   - **不要记**：能从代码 / git log / 当前文件状态推导的内容

3. **检索策略**：
   - 关键字匹配（fast，但召回低）
   - 向量相似度（语义匹配，但有噪声）
   - 混合检索（BM25 + 向量 + rerank）
   - **元数据过滤优先**：先按类型/时间/作用域过滤，再做语义检索

4. **遗忘与更新**：
   - 时间衰减：记忆带 `last_used_at`，长时间未用降权
   - 主动失效：当与新事实冲突时，覆盖或标记 stale
   - 版本化：重要记忆带版本号，避免"幻觉历史"

5. **本仓库的 memory 系统设计**：见 system prompt 中的 `auto memory` 段——一个文件即一条记忆，`MEMORY.md` 作索引，按 user/feedback/project/reference 四类组织。

### 实践任务
- [ ] 实现一个文件型 memory 系统：`memory/{type}_{topic}.md` + `MEMORY.md` 索引
- [ ] 加上向量检索：用 `sentence-transformers` 把每条 memory 嵌入，sqlite 存向量
- [ ] 设计一个"记忆冲突检测"工具：写入新记忆前，先检索相似的，问 LLM 是否要合并/覆盖
- [ ] 评测：让 agent 跨 5 个会话记住用户偏好，统计召回率

### 反模式警惕
- ❌ 把所有对话都写入"长期记忆"——会越变越垃圾
- ❌ 用一个超大的 memory 文件——读时全量加载，写时锁竞争
- ❌ 不带元数据的语义检索——容易召回不相关的旧信息

## W5 · 检索与 RAG

### 学习目标
- 理解 RAG 在 agent 中的定位：是"工具"而不是"架构"
- 掌握分块、嵌入、检索、重排、引用追溯五步流水线
- 能为代码库、PDF、网页三类语料分别设计检索方案

### 理论内容
1. **RAG vs Long Context**：
   - 长上下文模型（如 1M token）让简单 RAG 失去意义
   - 但 RAG 在 **延迟、成本、可追溯性** 上仍有优势
   - 决策标准：语料 < 上下文 30% 时塞进去；否则 RAG
2. **分块策略**：
   - 代码：按函数/类切分，保留头部 import 与签名
   - PDF：按章节 + 重叠窗口（512 token 块 + 64 token overlap）
   - 网页：按 HTML 结构（h1/h2/p）切，保留 DOM 路径
3. **嵌入选型**：
   - 通用：`text-embedding-3-large`、`bge-m3`
   - 代码：`voyage-code-2`、`jina-embeddings-v2-code`
   - 中文：`bge-large-zh`、`gte-large-zh`
4. **混合检索**：BM25（精确匹配）+ Dense（语义）→ Reciprocal Rank Fusion / Cross-encoder rerank
5. **引用追溯**：每个 chunk 带 source_id + offset，让 agent 输出时可以引用原文位置

### 实践任务
- [ ] 用 `chromadb` 或 `lancedb` 实现一个本仓库代码的语义搜索工具
- [ ] 加 BM25（用 `rank_bm25`），实现混合检索
- [ ] 接到 agent 上：当用户问"X 函数在哪里"，agent 自动调用检索工具
- [ ] 评测：构造 20 条 query，统计 top-5 命中率

### 进阶话题（W5 选学）
- **Agentic RAG**：让 agent 自己决定查询关键词、迭代检索、判断是否够答
- **HyDE**：先让 LLM 生成假设性答案，再用答案做检索
- **Query Rewriting**：把用户口语化的 query 改写成检索友好的形式

## W6 · 多智能体与子任务

### 学习目标
- 理解 subagent 的本质："上下文隔离的子调用"
- 掌握 orchestrator-worker、pipeline、debate 三种主流多智能体模式
- 能权衡"加 subagent" vs "加工具" vs "改 prompt"的取舍

### 理论内容
1. **何时用 subagent**：
   - 任务可以独立完成且产出可结构化总结
   - 主 agent 上下文需要被保护（避免被工具结果淹没）
   - 需要并行（多个独立子任务同时跑）
   - **反例**：紧密耦合的连续推理，不要拆——拆了反而丢上下文
2. **三种模式**：
   - **Orchestrator-Worker**：主 agent 派发，worker 干活，主 agent 汇总（Anthropic 的 Claude Code 模式）
   - **Pipeline**：A → B → C 顺序传递（适合 ETL 类任务）
   - **Debate / Reflect**：多个 agent 互相审稿（提高准确率，但成本翻倍）
3. **任务表示**：
   - 内存型：dict / 队列（简单但易丢）
   - 文件型：`tasks/*.json`（可持久化、可被多 agent 读写）
   - 数据库型：sqlite + 状态机（生产级）
4. **通信协议**：
   - 父子：参数 in，结构化总结 out
   - 兄弟：消息队列 / 共享文件
   - **避免共享可变状态**——和写多线程同样的坑
5. **失败处理**：超时、重试、降级（worker 失败时主 agent 是否自己干？）

### 实践任务
- [ ] 实现一个 `subagent` 工具：参数是 `prompt + tools`，内部跑一个独立的 agent loop，返回总结
- [ ] 让主 agent 把"读 10 个文件并总结"分派给 10 个并行 subagent（用 `asyncio` 或线程池）
- [ ] 加上文件型 task store（参考 `s07`），支持任务认领、完成、阻塞依赖
- [ ] 实验：debate 模式让 2 个 agent 互审一段代码，对比单 agent 的 bug 发现率

### 设计原则
> "Subagents are valuable for parallelizing independent queries or for protecting the main context window from excessive results, but they should not be used excessively when not needed."（来自 Claude Code 的设计哲学）

## W7 · 可靠性工程

### 学习目标
- 把 agent 从"能跑"推进到"能上线"
- 掌握评测集、重放、guardrails、可观测性四件套
- 能定位"为什么 agent 这次失败了"

### 理论内容
1. **评测（Eval）设计**：
   - 端到端评测：固定输入 → 期望输出，跑 100 次看通过率
   - 步骤级评测：每个工具调用是否合理、参数是否正确
   - LLM-as-Judge：用更强的模型给输出打分（注意 judge bias）
2. **轨迹（Trace）记录**：
   - 每次 LLM 调用的 input/output/latency/cost
   - 每次工具调用的 input/output/duration
   - 错误堆栈与重试次数
   - 工具：LangSmith、Phoenix、自研 sqlite + jsonl
3. **重放（Replay）**：
   - 录制真实流量，离线重放新版本 agent
   - 对比 trace 差异，找回归点
4. **Guardrails**：
   - 输入侧：PII 检测、prompt injection 检测
   - 输出侧：危险命令阻断、敏感信息过滤
   - 执行侧：工具调用频率限制、总成本上限
5. **常见故障模式**：
   - **工具循环**：反复调用同一工具——加迭代上限
   - **幻觉文件**：编辑不存在的文件——加预检查
   - **Prompt injection**：工具结果里夹带恶意指令——隔离/转义
   - **上下文污染**：早期错误信息一直留在 context——主动清理

### 实践任务
- [ ] 写一个 trace 装饰器，自动记录所有 LLM/工具调用到 `traces/{run_id}.jsonl`
- [ ] 构造 20 条 eval 用例（含正例、边界、对抗），跑通过率
- [ ] 实现 prompt injection 检测：当工具结果包含 "ignore previous instructions" 之类时告警
- [ ] 加 cost guard：单次会话超过 $0.5 自动停止
